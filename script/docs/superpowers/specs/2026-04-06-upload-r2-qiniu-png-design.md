# upload_r2.py 七牛云接入与 PNG 无损压缩设计

## 目标
在现有 `upload_r2.py` CLI 工作流中新增七牛云对象存储上传能力，并在上传前对 PNG 文件执行无损压缩。压缩结果需对 R2、Linux、七牛三个上传渠道同时生效，且全过程保持无界面、可自动化执行。

## 范围
- 为 `upload_r2.py` 增加七牛云上传目标。
- 在上传前新增统一的 PNG 无损压缩预处理层。
- 让同一份压缩后的临时文件同时用于 R2、Linux、七牛上传。
- 为七牛增加 skip-existing 与本地缓存支持。
- 扩展测试覆盖压缩、七牛上传、缓存与目标路由。

不在本次范围内：
- 修改 `upload_r2_gui.py`。
- 增加有损压缩、格式转换或非 PNG 重编码。
- 在上传失败时静默回退到未压缩原图。
- 打开任何外部图形界面或引入交互式工具流程。

## 方案选择
采用“统一上传前预处理层 + 七牛官方 Python SDK + oxipng CLI 临时压缩文件”的方案。

原因：
1. PNG 只需压缩一次，即可保证 R2、Linux、七牛三路行为一致。
2. `oxipng` 适合后台 CLI 集成，不会弹出界面，支持高压缩级别与临时输出文件。
3. 七牛官方 SDK 能直接完成上传与对象存在性检查，减少额外兼容层。
4. 预处理层可以把压缩逻辑与目标上传逻辑解耦，降低后续维护成本。

## 设计

### 1. 统一上传前预处理层
新增一个轻量预处理流程，例如 `prepare_upload_file(...)`，负责：
- 判断文件是否为 PNG。
- 非 PNG 直接返回原始路径与“未压缩”元数据。
- PNG 调用 `oxipng` 生成临时文件，并返回临时路径与“已压缩”元数据。
- 为调用方附带清理函数或临时文件上下文，确保上传后删除临时文件。

该层输出的不是“改写后的业务对象”，而是本次上传实际使用的本地文件路径和相关元信息，供三个上传渠道复用。

### 2. PNG 无损压缩策略
仅对 `.png` 生效，其他格式原样上传。

压缩工具固定为 `oxipng`，无界面命令行为：

```bash
oxipng -o max -z --strip safe --out <tmp> <src>
```

约束：
- 不启用 `--alpha`，避免引入视觉无损但字节级有变化的行为。
- 使用 `--strip safe`，仅移除安全可删除元数据。
- 使用临时输出文件，不覆盖用户原图。
- 若 `oxipng` 不存在、执行失败或未产出临时文件，则该文件直接记为失败，不回退原图上传。

### 3. 七牛上传实现
新增 `upload_to_qiniu(...)`，使用七牛官方 Python SDK：
- 通过 `qiniu.Auth(access_key, secret_key)` 生成认证对象。
- 通过 `upload_token(bucket)` 生成上传 token。
- 通过 `qiniu.put_file(...)` 上传文件。
- 通过 `BucketManager.stat(...)` 或等价元数据查询完成 skip-existing 在线检查。

对象 key 规则复用现有 `build_object_key(...)` 的思路，保证与 R2 目录结构一致。`prefix` 语义与 R2 对齐，统一映射为上传对象 key 前缀。

### 4. 目标类型与兼容策略
现有目标将从：
- `r2`
- `linux`
- `both`

扩展为：
- `r2`
- `linux`
- `qiniu`
- `all`

兼容策略：
- 继续接受 `both`，但内部映射到 `all`。
- `all` 表示 R2、Linux、七牛三路同时执行。
- 现有 `r2` 与 `linux` 行为保持不变。

### 5. 缓存与 skip-existing
缓存结构从当前的 `r2/linux` 扩展为 `r2/linux/qiniu`。

新增七牛缓存键：
- `bucket|object_key`

同时，文件指纹从当前“仅本地 size + mtime”扩展为可表达压缩策略的上传输入指纹，至少包含：
- 原始文件大小
- 原始文件修改时间
- 是否启用 PNG 压缩
- 压缩策略版本（例如 `oxipng:o_max:z:strip_safe`）

目的：
- 避免未来更改压缩策略后缓存误判“远端已是最新”。
- 保持三个目标的缓存行为一致。

七牛 skip-existing 逻辑与 R2/Linux 一致：
- 先查本地缓存命中。
- 缓存未命中时再进行远端在线存在性检查。
- 命中则跳过上传。

### 6. 上传函数边界
现有上传函数保留目标单一职责：
- `upload_to_r2(...)`
- `upload_to_linux(...)`
- `upload_to_qiniu(...)`

新增的预处理层位于它们之前。也就是说，上传函数不再各自决定是否压缩 PNG，而是统一接收“本次实际要上传的文件路径”。

为此会调整内部调用链，例如让 `upload_one(...)`：
1. 先生成对象 key / 远程路径。
2. 执行一次预处理。
3. 把预处理结果分发给目标上传函数。
4. 汇总三路结果。
5. 最后统一清理临时文件。

### 7. 错误处理
- PNG 压缩失败：该文件记 `failed`，错误直接暴露到日志。
- 单个目标上传失败：仅该目标记失败，不影响其他目标结果收集。
- `all` 模式下最终失败数为三路失败结果之和。
- skip-existing 在线检查失败：沿用当前脚本的错误返回风格，不增加复杂 fallback。
- 临时文件清理失败：记录日志但不覆盖主上传结果。

### 8. 配置来源
七牛配置继续沿用当前脚本的配置风格：CLI 参数优先，其次 `.env` / `upload_r2.env`。

计划支持的七牛配置：
- `QINIU_ACCESS_KEY`
- `QINIU_SECRET_KEY`
- `QINIU_BUCKET`
- `QINIU_PREFIX`
- `QINIU_DOMAIN`（如需要构造可读 URL 或日志输出）

为了保持与当前脚本一致，CLI 也会补充对应参数，例如：
- `--qiniu-bucket`
- `--qiniu-prefix`

密钥仍主要从环境变量读取，避免把敏感信息直接暴露在命令行历史中。

只有真正需要的项才做必填校验，避免引入无用配置项。

### 9. 并发策略
保持“上传并发”为主，不额外引入独立压缩任务队列。

策略：
- 每个文件在进入上传逻辑时内联执行一次 PNG 压缩。
- 一个文件只压缩一次。
- 压缩结果在当前文件的三个目标上传中复用。
- 继续使用现有并发模型调度不同文件，避免多层并发造成临时文件和资源管理复杂化。

### 10. 测试策略
补充最小但有针对性的回归测试：

1. 压缩预处理测试
   - PNG 会调用 `oxipng`。
   - 非 PNG 不调用压缩工具。
   - 压缩失败时返回失败结果。
   - 临时文件在上传结束后被清理。

2. 七牛上传测试
   - 成功路径：能调用官方 SDK 上传。
   - 失败路径：异常被转换为统一错误消息。
   - stat 检查能正确识别对象是否已存在。

3. 缓存测试
   - 七牛缓存命中时直接跳过。
   - 压缩策略变化时缓存失效。
   - `all` 模式下三路缓存都能正确更新。

4. 兼容性测试
   - `both` 能映射为 `all`。
   - `all` 会覆盖三个目标。
   - 现有 R2/Linux 测试不被破坏。

## 数据流
1. `main()` 解析 CLI 参数并加载环境变量。
2. `run_upload()` 收集文件、读取缓存、解析目标。
3. 每个文件进入 `upload_one()`。
4. `upload_one()` 调用预处理层：
   - PNG → `oxipng` 压缩到临时文件
   - 非 PNG → 原文件直传
5. 三个目标使用同一份实际上传文件：
   - R2 上传
   - Linux 上传
   - 七牛上传
6. 根据结果更新各自缓存。
7. 清理临时文件并输出最终统计。

## 风险
- `oxipng` 需要用户本机已安装；缺失时 PNG 上传会失败。
- `-o max -z` 追求高压缩率，处理速度会明显慢于普通模式。
- 七牛 SDK 的对象检查与上传返回结构需要在实现时用测试锁定，避免错误消息格式漂移。
- 现有缓存结构升级后，需要确保旧缓存文件仍能被安全读取并迁移到新结构。

## 成功标准
- `upload_r2.py` 支持 `qiniu` 与 `all` 目标。
- PNG 在上传前会通过 `oxipng -o max -z --strip safe` 压缩到临时文件。
- 同一份压缩结果同时用于 R2、Linux、七牛上传。
- 七牛支持 skip-existing 与本地缓存。
- 不修改 `upload_r2_gui.py`。
- 不打开任何外部图形界面。
- 至少有针对压缩、七牛上传、缓存与目标兼容的回归测试。
