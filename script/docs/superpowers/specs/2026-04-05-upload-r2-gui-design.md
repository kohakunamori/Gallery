# upload_r2 GUI 设计

## 目标
为当前 `upload_r2.py` 增加一个本地桌面 GUI，使用 PySide6 实现。界面顶部提供完整参数配置和启动按钮，底部左右分栏分别显示 R2 与 Linux 上传日志，同时复用现有上传核心逻辑，不重写上传实现。

## 范围
- 新增 PySide6 GUI 入口文件。
- 保留 `upload_r2.py` 作为核心上传逻辑。
- 为 GUI 增加适合调用的接口，使日志输出可注入回调而不是只能 `print`。
- 顶部参数区覆盖当前 CLI 主要参数。
- 底部双栏日志：左侧 R2，右侧 Linux。

不在本次范围内：
- 改变现有上传规则、对象 key 规则或 Linux 上传逻辑。
- 引入复杂主题、美化皮肤或托盘能力。
- 做复杂 GUI 自动化测试。

## 方案选择
采用“保留 `upload_r2.py` 核心逻辑 + 新增 `upload_r2_gui.py`”方案。

原因：
1. 当前上传逻辑已经可工作，GUI 应复用而不是重写。
2. CLI 与 GUI 分离，结构更清晰，避免把单文件脚本继续堆大。
3. 后续 CLI 与 GUI 可以并存，便于调试和回归验证。

## 设计

### 1. 文件结构
- `upload_r2.py`
  - 保留 CLI 入口和上传实现。
  - 提取/新增一个可供 GUI 调用的运行入口，例如接收配置对象或参数字典，并允许注入日志回调。
- `upload_r2_gui.py`
  - PySide6 GUI 入口。
  - 负责构建窗口、采集表单参数、启动后台线程、显示日志和状态。
- `tests/test_upload_r2.py`
  - 保留现有核心逻辑测试。
  - 补充少量针对 GUI 所依赖的新接口测试。

### 2. GUI 布局
采用用户选定的 B 方案：顶部按三组表单分区，底部双栏日志。

#### 顶部三组参数
1. **基础参数**
   - `dir`
   - `target`
   - `recursive`
   - `workers`
   - `dry_run`
   - `skip existing`（通过现有 `no-skip-existing` 语义映射）
   - `env-file`
   - `启动` 按钮

2. **R2 参数**
   - `bucket`
   - `prefix`
   - `endpoint`
   - `region`

3. **Linux 参数**
   - `linux_host`
   - `linux_user`
   - `linux_dir`
   - `linux_key`
   - `linux_password`
   - `linux_port`

#### 底部日志区
- 左侧文本面板：R2 日志
- 右侧文本面板：Linux 日志
- 顶部再加一个简短状态栏，用于显示“空闲 / 运行中 / 完成 / 失败”

### 3. 运行模型
GUI 点击“启动”后：
1. 从表单读取参数。
2. 构造后台 worker 线程。
3. 在线程中调用 `upload_r2.py` 暴露的可调用接口。
4. 核心逻辑每产生日志，交给 GUI 注入的回调。
5. GUI 根据日志前缀路由：
   - `[R2]` → 左侧日志框
   - `[LINUX]` → 右侧日志框
   - 汇总/状态信息 → 状态栏，必要时也可同时追加到对应日志区
6. 任务运行期间禁用“启动”按钮，结束后恢复。

### 4. 核心逻辑适配
当前 `main()` 深度依赖 `print` 和 `argparse`。为了支持 GUI，需要增加一个面向调用方的接口，负责：
- 接收已经解析好的参数，而不是只从命令行读取。
- 在每次状态输出时走 `log_callback(message)`，默认仍可回退到 `print`。
- 返回最终退出码 / 运行结果，供 GUI 更新状态栏。

关键点：
- CLI 仍保留，继续可通过 `python upload_r2.py ...` 运行。
- `main()` 可以继续存在，但内部改为复用新的可调用入口。
- 不改变实际上传分支（R2 / Linux / both）的业务逻辑。

### 5. 错误处理
- GUI 只做表单层基础校验，例如数值字段是否可解析。
- 上传线程里的异常需要捕获，并把失败信息显示到状态栏和相应日志区。
- 不增加额外业务 fallback，仍以核心逻辑现有规则为准。

### 6. 测试策略
1. 保留现有核心上传测试。
2. 为新增的“GUI 可调用接口”写最小测试，确认：
   - 可通过回调接收日志
   - 返回码与 CLI 行为一致
3. GUI 界面本身不做重型自动化测试，本次只验证：
   - 能启动窗口
   - 能构造参数
   - 能正确路由 R2 / Linux 日志

## 数据流
1. 用户在 GUI 输入参数。
2. GUI 组装配置并启动后台线程。
3. 后台线程调用 `upload_r2.py` 的运行入口。
4. 运行入口执行上传，并把日志逐条交给 GUI 回调。
5. GUI 分流日志到左右面板，并更新状态栏。

## 风险
- 需要新增 `PySide6` 依赖，安装体积较大。
- 现有脚本输出路径较多，若不先抽出统一日志接口，GUI 接入会变得零散。
- 后台线程与 UI 更新之间需要通过 Qt signal/slot 安全通信，不能直接跨线程改控件。

## 成功标准
- 可以直接运行一个 PySide6 GUI 程序。
- 顶部有完整参数区和启动按钮。
- 底部左侧显示 R2 日志，右侧显示 Linux 日志。
- 上传任务在后台线程执行，不阻塞界面。
- CLI 入口继续可用。
- 现有核心测试继续通过，并补充最小的新接口测试。
