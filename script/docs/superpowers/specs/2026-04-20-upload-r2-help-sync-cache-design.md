# upload_r2 CLI 帮助信息整理与缓存对齐模式设计

## 背景

当前 `upload_r2.py` 的命令行帮助信息为英文且未分组，阅读成本较高。同时，现有缓存逻辑虽然支持在上传前按需远端预检查，但缺少一个“仅根据远端真实状态重建本地缓存、不执行上传”的独立模式。除此之外，上传模式当前会逐条输出 `SKIP` 项，日志噪声偏大。

本次改动目标是：

1. 整理并汉化 `--help` 输出；
2. 在不改变现有参数名与上传行为兼容性的前提下，新增仅同步缓存的执行模式；
3. 在上传模式下保留跳过统计，但不逐条输出 `SKIP` 项目。

## 目标

### 目标 1：整理并汉化 `--help`

将 CLI 帮助信息调整为中文，并按职责分组展示，至少分为：

- 通用参数
- R2 参数
- Linux 参数
- 七牛参数

要求：

- 保持现有参数名不变；
- 保持默认值与行为不变；
- 仅提升帮助信息可读性与信息组织结构。

### 目标 2：支持“仅对齐本地缓存和远端状态”

新增独立参数 `--sync-cache-only`，用于：

- 扫描本地图片；
- 按 `--target` 对应的目标端检查远端对象/文件是否存在；
- 依据远端实际存在状态，重建本地缓存；
- 不上传任何文件。

这里的“重建缓存”语义为：

- 若远端存在：为对应 target 写入或更新缓存记录；
- 若远端不存在：删除该 target 在本地缓存中的旧记录；
- 仅影响当前 `--target` 覆盖到的目标缓存，不应误删其他 target 的缓存记录。

## 非目标

以下内容不在本次范围内：

- 修改现有参数名；
- 修改默认上传目标、上传策略或压缩策略；
- 重写缓存文件格式；
- 新增自定义 formatter 之外的大规模 CLI 框架替换；
- 对未选中的 target 做缓存清理；
- 隐藏上传成功或失败日志，只有 `SKIP` 逐条日志会被收敛。

## 方案比较

### 方案 A：仅汉化帮助信息

优点：改动最小。

缺点：无法满足新增缓存对齐能力。

### 方案 B：汉化 + 参数分组 + 新增 `--sync-cache-only`（推荐）

优点：

- 语义清晰；
- 与现有 `--dry-run`、`--verify-remote` 的职责边界明确；
- 对现有上传流程侵入较小。

缺点：

- 需要在主流程中新增一条独立分支；
- 需要补充删除 target 缓存记录的辅助逻辑。

### 方案 C：把缓存对齐塞进 `--verify-remote`

优点：少一个参数。

缺点：

- `--verify-remote` 现有语义是“仅对本地待上传集合做远端确认”；
- 若扩展为“全量重建缓存”，行为会变得含混且不兼容用户直觉。

结论：采用 **方案 B**。

## `--sync-cache-only` 行为设计

### 总体行为

当传入 `--sync-cache-only` 时，程序进入“仅同步缓存”模式：

1. 读取环境变量与参数；
2. 扫描本地图片；
3. 加载或重建本地缓存；
4. 针对选中的 target 全量检查远端存在状态；
5. 重建本地缓存；
6. 输出统计并退出；
7. 不执行任何上传动作。

### 与其他参数的关系

#### `--dry-run`

与 `--sync-cache-only` 互斥。因为两者都表示“不上传”，但一个强调“预览”，一个强调“实际改写缓存”，语义不同。

#### `--verify-remote`

在 `--sync-cache-only` 模式下无意义。因为该模式本身就是对选中 target 做全量远端校验。

实现上建议直接报错提示，而不是悄悄忽略，以避免歧义。

#### `--no-skip-existing`

在 `--sync-cache-only` 模式下无意义。因为没有上传动作，也不存在“跳过已存在对象”的执行分支。

实现上建议直接报错提示。

#### `--refresh-cache`

可以保留，并与 `--sync-cache-only` 配合使用：

- 若指定 `--refresh-cache`，先清空缓存再按远端状态重建；
- 若未指定，则在现有缓存上按 target 维度重建所选目标的状态。

最终结果对选中 target 来说应等价于重建后的正确状态。

## 缓存重建设计

### 需要补充的能力

当前代码已有“写入/更新 target 缓存”的能力，但缺少“按 target 删除缓存记录”的能力。

因此需要新增一个辅助函数，语义类似：

- 删除某个相对路径下指定 `target_label` 的缓存记录；
- 若 `targets` 为空则移除空字典；
- 若整个文件记录为空，则移除该文件记录；
- 不影响同一路径下其他 target 的缓存记录；
- 不破坏 `source`、`prepared_png` 等与其他 target 仍然相关的数据，除非该记录已经整体为空。

### 重建流程

对于当前 `--target` 选中的每个 target：

1. 基于全部本地文件计算远端标识；
2. 查询远端实际存在集合；
3. 遍历全部本地文件：
   - 若存在于远端：更新 target 缓存；
   - 若不存在于远端：删除该 target 缓存；
4. 如果缓存有变化，则标记 `cache_dirty=True`。

### 各 target 的存在性判断

#### R2

基于对象 key 全量确认远端是否存在。优先复用现有 `list_existing_keys(..., object_keys=...)`。

#### Linux

基于当前已有的 Linux 远端检查逻辑，对本地全部文件进行存在性确认。可复用：

- 按 basename 的快速命中；
- 逐文件 SSH/SFTP 远端判断。

需要保证该逻辑在“仅同步缓存”模式下返回的是完整存在集合，而不是仅返回 skip 结果文本。

#### 七牛

基于对象 key 全量确认远端是否存在。优先复用现有 `list_existing_qiniu_keys(...)`。

## 运行输出

### `--sync-cache-only` 模式

在 `--sync-cache-only` 模式下，输出应区别于上传模式，至少包括：

- 扫描到的本地文件数；
- 当前 target；
- 每个 target 远端确认存在数；
- 每个 target 新增缓存数；
- 每个 target 删除缓存数；
- 每个 target 保持不变数；
- 失败数；
- 最终总结：缓存同步完成。

避免输出“Uploaded / Skipped”这类上传语义词汇。

### 普通上传模式

普通上传模式下：

- 仍保留最终汇总中的 `Skipped` 计数；
- 仍允许内部以 `skipped` 状态驱动缓存更新；
- 但不再逐条输出 `[TARGET] SKIP ...` 日志；
- `uploaded` 与 `failed` 的逐条日志保持不变。

这样可以保留统计与缓存语义，同时减少大量重复跳过日志造成的噪声。

## 测试设计

遵循 TDD，先补失败测试，再实现代码。

至少覆盖以下场景：

1. `--sync-cache-only` 运行时不触发上传函数；
2. 远端存在对象时，会为对应 target 写入缓存；
3. 远端不存在对象时，会删除该 target 的旧缓存；
4. 删除某 target 缓存时，不影响其他 target 缓存；
5. `--sync-cache-only` 与 `--dry-run` 同时传入时，返回参数错误；
6. `--sync-cache-only` 与 `--verify-remote` 同时传入时，返回参数错误；
7. `--help` 中出现中文说明和分组标题；
8. 普通上传模式下遇到 `skipped` 结果时，不输出逐条 `SKIP` 日志，但最终统计仍包含 `Skipped` 数量；
9. `--sync-cache-only` 模式下的缓存对齐结果仍会输出自己的汇总日志。

若项目当前没有完整测试框架，可先添加最小可运行的 Python 测试，覆盖核心缓存重建逻辑与参数校验逻辑。

## 风险与缓解

### 风险 1：误删其他 target 缓存

缓解：删除函数必须以 `target_label` 为边界，仅删除当前目标对应记录，并补测试覆盖“保留其他 target”。

### 风险 2：Linux 存在性检查成本较高

缓解：优先复用已有批量/预检查逻辑，不额外设计新协议；先保证语义正确，再考虑性能。

### 风险 3：帮助信息改动影响既有使用习惯

缓解：参数名保持不变，只调整展示顺序与中文说明。
